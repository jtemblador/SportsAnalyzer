Function 5: Feature Engineering Pipeline

The feature engineering system transforms raw player statistics into predictive
features suitable for machine learning. This includes calculating rolling averages, opponent
strength adjustments, and usage trends.

Implementation Details:

The FeatureEngineer class processes each week of historical data sequentially. For

each player in a given week, it loads their previous six games using the load_player_history
method, which searches backwards through weeks and crosses into previous seasons if
necessary. It then calculates weighted rolling averages using exponential decay, where the
most recent game has a weight of 1.0, the previous game 0.9, the one before that 0.81, and so
on. This gives more importance to recent performance while still considering historical
context.

The system also calculates opponent strength by analyzing how many fantasy points

each defense has allowed to a specific position throughout the season. A defense ranked 32nd
is the easiest matchup, while rank 1 is the toughest. Usage trends track whether a player's
involvement is increasing or decreasing by comparing their last three games to their previous
three games in metrics like pass attempts, targets, or carries.

Current System Architecture

The system follows a modular architecture with clear separation of concerns. The

NFLDataPipeline class handles all data acquisition and storage, never touching presentation
logic. The Streamlit dashboard handles user interface and display, delegating all data
operations to other components. The FeatureEngineer class transforms raw statistics into
machine learning features, operating independently of both the pipeline and dashboard.

Data flows through the system in a unidirectional pipeline. Raw data enters through

the NFLDataPipeline via the nflverse API and gets stored as parquet files in data/nfl/raw. The
FeatureEngineer reads these raw files, processes them, and outputs engineered features to
data/nfl/cleaned. Future machine learning models will read from the cleaned directory, train
on historical features, and output predictions to data/nfl/predictions.

This architecture provides several benefits. Each component can be tested

independently. The pipeline can be run from the command line without starting the
dashboard. Features can be re-engineered with different parameters without re-downloading
raw data. New machine learning models can be trained without modifying the feature
engineering code. The system remains maintainable and extensible as complexity grows.

Data Coverage and Statistics

The system currently maintains comprehensive coverage of NFL player statistics. The

raw data directory contains 108 parquet files representing every week from the 2020 season
through Week 11 of the 2025 season. Each file averages 2,500 player records with 114
statistical columns, totaling approximately 270,000 individual player-game combinations.

The cleaned data directory now contains engineered features for these same weeks.

Each feature file includes 20 to 30 calculated metrics per player depending on their position.
Quarterbacks have passing-specific features like rolling average passing yards and opponent
pass defense rank. Running backs have rushing and receiving features. Wide receivers and
tight ends have target and reception metrics. This position-specific approach ensures that
each machine learning model receives relevant features for its prediction task.

Performance Characteristics

The system demonstrates strong performance across all operations. Data fetching for a

single week completes in 3 to 5 seconds depending on network speed. The incremental
update check that scans existing files takes less than 100 milliseconds even with 108 files.

Feature engineering represents the most computationally intensive operation.

Processing a single week with 2,500 players takes approximately 50 to 60 seconds. This
includes loading history for each player, calculating rolling averages, determining opponent
ranks, and writing the output file. Processing all 108 weeks requires approximately an hour,
but this is a one-time operation. Subsequent runs skip existing weeks and only process new
data.

The Streamlit caching system provides additional performance benefits. When a user
first selects a season, the load_season_data function reads and concatenates 18 parquet files.
This takes roughly 800 milliseconds. However, Streamlit caches the result using the
st.cache_data decorator. If the user changes filters but keeps the same season, the data loads
from cache in under 10 milliseconds. This creates a responsive user experience where filter
changes feel instantaneous.

Technical Decisions and Rationale

Several key technical decisions shape the system's architecture and capabilities. The

choice to use parquet files instead of a traditional database reflects the analytical nature of the
workload. Queries typically request a few columns across many rows, which is exactly what
columnar storage optimizes for. Parquet also provides built-in compression and fast
serialization, making it ideal for time-series data like weekly player statistics.

The local-first architecture prioritizes reproducibility and user control over cloud

convenience. Researchers can archive the entire data directory and reproduce analyses years
later with confidence that data has not changed. No API keys need management, no rate
limits require workarounds, and no subscription fees accumulate over time. The system runs
entirely offline once data is downloaded.

The decision to calculate features with exponential decay rather than simple averages

reflects the reality that recent performance predicts future performance better than distant
history. A player who scored 30 points last week is more likely to score well this week than a
player whose 30-point game was six weeks ago. The decay factor of 0.9 balances recency
with stability, preventing single-game outliers from dominating the average while still
weighting recent trends heavily.

3.2 NFLDataPipeline Class

The NFLDataPipeline class serves as the core component responsible for all data

acquisition and storage operations. This class encapsulates the entire process of fetching data
from the nflverse API and organizing it into the local file system. The three field variables
define the directory structure where data is stored. The base_data_dir field defaults to "./data"
and serves as the root directory for all sports data. The nfl_dir field combines the base
directory with "/nfl" to create a sport-specific folder, and the raw_dir field adds "/raw" to
store unprocessed data directly from the API.

The initialization method creates these directories if they do not already exist using

Python's pathlib library. This ensures the system can run on a fresh installation without
manual directory creation. The get_current_week and get_current_season methods query the
nflverse API to determine what week and season are currently active in the real NFL

calendar. These methods are useful for validation but are not currently used in the automated
update process.

The get_last_downloaded_week method implements the logic for incremental

updates. It scans all parquet files in the raw directory, extracts season and week numbers
from filenames using string splitting, and returns the maximum combination found. If no files
exist, it returns season 2024, week 18 as a starting point. The check_file_exists method
constructs the expected filename for a given season and week, then checks if that file exists in
the file system. This prevents redundant downloads.

The fetch_player_stats method is where data actually comes from the external API. It

accepts a list of seasons and an optional week number, calls the nflreadpy library's
load_player_stats function, converts the resulting Polars dataframe to pandas for consistency,
and filters to the specific week if one was provided. The save_data method takes a dataframe
and writes it to a parquet file with compression, using the standardized naming convention. It
returns the full file path for confirmation.

The run_pipeline method orchestrates the entire fetch-and-save process. It first checks
if the data already exists to avoid duplicates, then fetches the data, validates that records were
returned, prints a summary of what was downloaded, and saves it to disk. The silent_check
parameter allows suppressing output messages when called in a loop. This method serves as
the main entry point for both manual data fetching and automated updates.

3.4 FeatureEngineer Class (Phase 3 - Not Yet Built)

The FeatureEngineer class will be built during Phase 3 to transform raw statistics into
features suitable for machine learning. Raw statistics like "rushing yards" or "receptions" are
useful for analysis but insufficient for prediction because they lack context. Feature
engineering adds that context.

The decay_factor field will control how much recent games are weighted compared to

older games when calculating rolling averages. A decay factor of 0.9 means the most recent
game contributes 100% of its value, the previous game contributes 90%, the game before that
81%, and so on. The window_size field will determine how many games to include in rolling
calculations.

The calculate_rolling_average method will take a player's historical data and compute

weighted averages that give more importance to recent performance. The
adjust_for_opponent method will modify baseline expectations based on opponent strength,
recognizing that 100 rushing yards against the best run defense in the league is more

impressive than 100 yards against the worst. The calculate_usage_metrics method will
compute what percentage of team plays involve a specific player, helping distinguish
between a running back who gets 20 carries per game and one who gets 5.

4.2 Data Storage Architecture

The system uses parquet files instead of a traditional relational database. Parquet is a

columnar storage format originally developed for big data systems like Apache Spark. Unlike
CSV files that store data row by row, parquet organizes data by columns, which makes it
extremely efficient for analytics where you typically want to read a few columns across many
rows rather than reading entire rows.

All data files live in the ./data/nfl/raw/ directory with a strict naming convention.

Every file follows the pattern player_stats_{SEASON}week{WEEK}.parquet, so the file for
week 5 of the 2025 season is player_stats_2025_week_5.parquet. This naming scheme makes
it trivial to find data for any specific week and easy to write loops that process entire seasons.
The raw subdirectory indicates that this data has not been modified from what the API
provided, maintaining full transparency about data provenance.

6.2 Next Steps and Timeline

Phase 3 will implement machine learning models during weeks 7 through 9. The POB

model (Probability Over Baseline) will be a binary classifier that predicts whether a player
will exceed their historical average performance. The EVOB model (Expected Value Over
Baseline) will be a regression model that predicts by how much a player will exceed or fall
short of their baseline. Both models will use gradient boosting algorithms, specifically
XGBoost or LightGBM, which excel at handling tabular data with complex interactions
between features. Walk-forward validation will ensure that models are only trained on past
data and evaluated on future data, preventing the look-ahead bias that plagues many sports
prediction systems.

the OpenAI GPT API will translate SHAP values into natural language explanations like
"The model predicts Tyreek Hill will exceed his baseline this week primarily because he
faces a weak pass defense and has been targeted heavily in recent games." Final dashboard
polish will improve the visual design, add export capabilities, and incorporate user feedback
from testing.

KEY TECHNICAL DECISIONS
================================================================================

1. FILE NAMING CONSISTENCY
   Decision: All V4 files use v4_* prefix
   Reason: Easy identification of version-specific files
   Impact: Better file organization and clarity

2. CACHE ORGANIZATION
   Decision: Subdirectories (team_lines/, player_props/) with week/date in filename
   Reason: Clear, intuitive, easy to locate specific data
   Example: team_lines_week_14_fetched_2025-12-06.parquet

3. SMART CACHING STRATEGY
   Decision: Cache-first approach, fetch only missing data
   Reason: Preserve API credits (500/month limit)
   Impact: Test showed 0 API calls when data cached
   Workflow: Check cache → Only fetch missing → Update incrementally

4. RETRAIN SCRIPT STANDARDIZATION
   Decision: All versions use identical auto-detection pattern
   Reason: Consistency, maintainability, future-proof
   Impact: Created missing v1_retrain.py, updated v2/v3/v4

TECHNICAL ARCHITECTURE
================================================================================

V4 VEGAS ODDS WORKFLOW:
1. Run predictions → Identify top 20-30 promising players
2. Call fetch_props_for_specific_players(player_names, week)
3. System checks cache first
4. Only fetches missing players from API
5. Updates cache incrementally
6. Returns combined cached + new data
7. Zero API calls if all players cached

RETRAIN SCRIPTS AUTO-DETECTION PATTERN:
```python
raw_data_dir = project_root / 'data/nfl/raw'
available_weeks = []

for week in range(1, 19):  # Check weeks 1-18
    raw_file = raw_data_dir / f'player_stats_2025_week_{week}.parquet'
    if raw_file.exists():
        available_weeks.append(week)

# Generate predictions for ALL found weeks
for week in available_weeks:
    predictions = pipeline.generate_predictions(2025, week)

# Validate on last 3 weeks
validation_weeks = available_weeks[-3:]
```

================================================================================
UNDERSTANDING: VEGAS LINES VS PLAYER PROPS
================================================================================

Vegas Lines (Team-Level):
- Point spread (favorite/underdog)
- Over/under (total game points)
- Implied team totals
- Use case: Game context for prediction models
- Predict volume/opportunity

Player Props (Player-Level):
- Individual player lines (passing yards, rushing yards, receptions, etc.)
- Use case: Compare model predictions vs Vegas
- Validate prediction accuracy
- Identify edges/opportunities

Workflow:
1. Train models with Vegas lines as features (game context)
2. Generate predictions for upcoming week
3. Fetch player props for top 20-30 predictions only
4. Compare model predictions vs Vegas lines
5. Identify where model differs from Vegas (potential edges)

LESSONS LEARNED
================================================================================

1. WHY V3 FAILED (EPA Features)
   - Added player-level efficiency metrics (EPA, CPOE, etc.)
   - Problem: Efficiency doesn't predict volume
   - Example: Efficient player with 2 targets < Inefficient with 10 targets
   - Result: 0% improvement over V2

2. WHY V4 SHOULD WORK (Game Context)
   - Adding game-level context features (Vegas lines)
   - Vegas lines predict volume/opportunity
   - Spread predicts game script (passing vs rushing)
   - Total predicts plays/pace (more opportunities)
   - Research shows 80%+ accuracy for totals

3. SMART API USAGE
   - Free tier: 500 requests/month
   - Cache-first approach prevents waste
   - Only fetch missing players (not all)
   - Test showed: 0 API calls when cached
   - Strategy makes 500 requests sufficient

4. CONSISTENCY MATTERS
   - All retrain scripts now follow same pattern
   - Makes maintenance easier
   - Reduces bugs from inconsistency
   - Created missing v1_retrain.py for completeness

Task 0.2 — Injury Fetcher (Weekly Injury Reports + Practice Status)
Date: 2026-04-10
Status: Complete
Tests: 15 new (50 total across project)

Goal
-----
Pull weekly NFL injury report data from nflreadpy for seasons 2018-2025.
This gives us game-day status (Out/Doubtful/Questionable), primary injury
type, and practice participation (Full/Limited/DNP) — data that directly
impacts player availability and fantasy scoring predictions.

Approach
--------
- Used nfl.load_injuries() which returns 16 columns per player per week
  including report_status, report_primary_injury, and practice_status.
- One Parquet file per season: data/nfl/injuries/injuries_{season}.parquet
- Stored data as-is with no transformations. Nulls in report_status are
  intentional — they indicate the player was not on the injury report (healthy).
- Built InjuryFetcher following the exact same pattern as ScheduleFetcher
  (Task 0.1) — standalone class registered in NFLDataPipeline.

Validation
----------
- Spot-check: Mahomes 2024 Week 18 → report_status "Doubtful", injury "Ankle",
  practice_status "Limited Participation in Practice". Correct.
- gsis_id: 0 nulls across all seasons (clean join key to player stats).
- report_status values: only Out/Doubtful/Questionable/Note (no unexpected values).
- practice_status: 99%+ populated across all seasons.
- Row counts: 5,133-6,215 per season (reasonable, ~1,300-1,500 unique players).
- All 8 seasons (2018-2025) fetched successfully.

Lessons Learned
---------------
- report_status is ~54% null — this is NOT missing data. Players only appear on
  the injury report if they have an injury. Null = healthy/available. When building
  features, null report_status should be treated as "active/available."
- practice_status is the most reliable signal (~99% populated). report_status is
  only assigned on the final injury report before game day.
- report_secondary_injury is 97% null — most injuries are single. Keep column
  but don't rely on it for features.
- practice_status has 36 rows with whitespace-only values ('\n    ') in 2024.
  Clean these during feature engineering, not in raw data storage.
- The fetcher pattern (standalone class + pipeline registration) scales cleanly.
  Two fetchers now in pipeline, 11 more to add in Tasks 0.3-0.12.

Task 1.3 — Bulk Load All Parquet Data into PostgreSQL
Date: 2026-04-11
Status: Complete
Tests: 14 total in test_database.py (rewritten for post-load state)

Goal
-----
Load all 14 Parquet datasets into PostgreSQL so historical data (2018-2025)
is queryable via SQL. 798,176 rows across 14 tables.

Approach
--------
- Built load_all.py with declarative TABLES config — each entry defines
  table name, data directory, file pattern, season range, and optional filter.
- Uses pandas to_sql() with method='multi' and chunksize=5000 for batch inserts.
- Auto-truncates existing data if tables are non-empty (idempotent reloads).
- Filters null player_id garbage rows from weekly_stats (173 rows removed).

Key Issue Discovered: Cross-Season Column Mismatches
----------------------------------------------------
The schema generator originally used a single season's Parquet to define columns.
Three datasets have columns that only appear in certain seasons:
- player_stats: 'game_id' only in 2022, 2023, 2025 (not 2018-2021, 2024)
- injuries: 'season_type' only in 2025 (not 2018-2024)
- team_stats: 'game_id' only in 2022, 2023, 2025

Fixed by updating generate_schema.py to load ALL season files and take the
column union. Schema now includes every column from any season.

Lessons Learned
---------------
- Cross-season column differences are a real problem. Always generate schemas
  from the union of ALL seasons, not a single representative year. nflverse
  adds new columns over time (game_id added ~2022, season_type added 2025).
- pandas to_sql() with method='multi' handles missing columns gracefully —
  seasons without a column just have NULL in those rows.
- Bulk load takes ~10 minutes for 798K rows. Acceptable for a one-time load.
  Incremental updates (Task 1.4) will be much faster.
- Tyler Bass (K) shows 0 fantasy_points_ppr despite 2 FG + 4 XP — this is
  because nflreadpy's fantasy_points_ppr uses standard scoring which doesn't
  count kicker points. Kicker scoring is separate. Not a bug.

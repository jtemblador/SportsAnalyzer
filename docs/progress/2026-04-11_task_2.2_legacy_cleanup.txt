Goal
-----
Organize the codebase so active code is clearly separated from historical
ML reference code. Delete truly dead code, preserve V1-V4 ML progression
in a dedicated legacy/ directory, and leave a clean test suite with 0 failures.

Approach
--------
Used a hybrid strategy instead of pure deletion:
1. Tagged current HEAD as v4-final for full reproducibility
2. Moved V1-V4 ML code to legacy/v1-v4/ — preserves the ML progression
   story (5.14→4.26 MAE) visually in the repo for portfolio reviewers
3. Deleted truly dead code (paid Odds API) and broken tests
4. Updated smoke tests to reflect active modules only

This approach was chosen because:
- Pure deletion loses the ML journey narrative (important for portfolio)
- Pure "keep everything" clutters active code with dead references
- The hybrid shows progression while keeping src/ clean

Lessons Learned
---------------
- For ML portfolio projects, a legacy/ directory is better than pure deletion.
  It tells the story: "I iterated through 4 versions, improving accuracy 17%."
  Recruiters can browse the progression without needing git commands.
- Git tags (v4-final) provide the safety net. If we ever need to reproduce
  V4 results exactly, the full codebase is one command away.
- Broken tests that aren't collected by pytest (import failures at module
  level vs function level) are invisible — they don't show as failures but
  they also don't test anything. Better to delete them than leave phantom
  coverage.

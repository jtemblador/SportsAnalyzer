Goal
-----
Load all V1-V4 prediction data into PostgreSQL and backfill actual results
from weekly_stats so we can run cross-version accuracy queries in SQL.

Approach
--------
- Created two new tables: model_versions (metadata) and predictions (results).
- Built an ETL script that scans prediction directories, loads Parquet files,
  and backfills actual values by joining with weekly_stats per stat.
- Predictions table includes computed error column (abs difference).
- Review caught an FK ordering bug: on re-runs, DELETE FROM model_versions
  would fail because predictions still referenced it. Fixed by truncating
  predictions first in the ETL flow.

Lessons Learned
---------------
- DB-computed MAE (4.67 for V4) differs from reported training MAE (4.26)
  because training MAE comes from walk-forward validation (train/test split)
  while DB MAE is computed on all 2025 predictions with actuals. The relative
  ordering is preserved — V4 is consistently best. Both numbers are valid,
  they just measure different things.
- FK ordering matters for idempotent ETL: must truncate the child table
  (predictions) before deleting from the parent (model_versions). The review
  phase caught this before it became a production issue.
- POB model rows have NULL predicted_value (they predict probability, not
  a stat value). These are correctly excluded from MAE calculations via
  WHERE predicted_value IS NOT NULL.

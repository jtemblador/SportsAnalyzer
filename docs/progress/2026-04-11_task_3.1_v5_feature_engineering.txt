Goal
-----
Build src/nfl/features/v5/ package that transforms raw NFL Parquet data into
60-80 predictive features per player per week, feeding the V5 model training
(Task 3.2). Must run on Google Colab (reads Parquet, not PostgreSQL) to
leverage high-RAM cloud CPUs for the 8-season batch.

Approach
--------
1. Package structure, not monolith: 7 focused files (config, master_table,
   rolling, context, usage, advanced, engineer). Each < 300 lines. V4's
   legacy/v1-v4/features/v4_engineer.py was one 600-line monolith — we
   split to keep each concern independently testable.

2. Master player-week table approach: Build one DataFrame via 13 LEFT JOINs
   (schedules, injuries, snap_counts, NGS passing/rushing/receiving, PFR
   pass/rush/rec, FF opportunity, depth_charts, team_stats, players). Each
   feature module then computes derived features from this single table,
   using vectorized pandas operations.

3. GSIS ↔ PFR ID mapping once at the join level: snap_counts and pfr_advstats
   use PFR IDs; everything else uses GSIS. The master table joins through
   the players table once so downstream feature code is ID-agnostic.

4. Subagent-driven TDD build: 8 bite-sized tasks, each with write-failing-
   test → implement → verify-pass → commit. Fresh subagent per task to
   preserve context and enable parallel review.

5. Two review passes: First pass caught FK ordering + kicker pipeline gap.
   Second pass caught 4 more (position filter, games_of_history, feature
   groups, schema consistency). All fixed before handoff.

Key V5 decisions (from docs/V5_QUESTIONS.md)
--------------------------------------------
- Predict individual stats per position (not fantasy_points_ppr directly).
  Dashboard shows per-stat over/under. Fantasy points computed from sum.
- 2 model types (StatPredictor + POB). Dropped EVOB (redundant).
- Pre-game features only. No current-week data used (strict shift/expand).
- Parquet-first, no DB dependency. Runs on Colab without PostgreSQL.
- Cross-season rolling is INTENTIONAL. 2018-2019 warm-up data so week 1
  of 2020 has a full lookback window. Player identity carries across
  season boundaries.
- Position filter at master-table level: drop LB/CB/DL/OL/etc. (65% of
  rows). Only QB/RB/WR/TE/K enter the feature pipeline.
- NULL preservation for NGS/PFR coverage gaps. Tree-based models handle
  NULLs natively — imputing 0 would lie about what we know.

Lessons Learned
---------------
1. Cross-season rolling is intentional, not leakage. The first review pass
   flagged groupby('player_id') as a bug. Deeper reading of V5_ROADMAP
   revealed 2018-2019 are explicitly warm-up seasons to enable week 1 of
   2020 to have a full lookback window. Documented in rolling.py + test.

2. Review passes catch real bugs. First pass missed position filter (65%
   row bloat), games_of_history (enabling column for MIN_GAMES_HISTORY
   downstream), and FEATURE_GROUP_PREFIXES (ablation study enabler).
   Second pass took a different angle (holistic integration) and caught
   all three. Always do two review passes on large tasks.

3. Pyright warnings are noise on pandas-heavy code. All 50+ diagnostic
   warnings during this task were false positives — pandas type inference
   is imperfect for .rename/.drop_duplicates/.merge. Runtime correctness
   is what matters; unit tests are the ground truth.

4. Master-table approach beats per-player loops. V4 loaded Parquets per
   player per week (slow). V5 loads all seasons once, joins into master
   table, runs vectorized groupby.shift(1).expanding().mean() for rolling
   features. Faster and easier to reason about — leakage protection is
   enforced at the shift() level, not spread across the codebase.

5. Feature group prefixes > feature group categories. Ablation study
   needs to drop columns by group. Hardcoding column lists is brittle
   (will drift as features are added/removed). Using prefix patterns
   (rolling_avg_*, ngs_*, pfr_*) makes the grouping self-maintaining.

6. Injury dedup policy matters. A player can have multiple injury report
   updates in one week (DNP Wednesday → Limited Thursday → Questionable
   Friday). keep='last' preserves the final game-day status, which is
   what the model should use. Documented the decision in master_table.py.

7. NGS/PFR column inclusion must be selective. Raw NGS has 29 columns;
   including all would bloat the feature set without adding signal. We
   pick 3-4 columns per NGS/PFR source (time_to_throw, CPOE, aggressiveness
   for passing; separation, YAC above expected, cushion for receiving).
   Ablation study (Task 3.2b) will validate these choices.

Remaining Risks / Deferred
--------------------------
- MIN_GAMES_HISTORY = 3 is defined but not enforced in the feature engineer.
  games_of_history column now exists so Task 3.2 (training) can filter on it.
- Rolling loop is Python-level O(players × weeks × stats). Functional but
  may be slow on large runs. Optimize only if Colab run is too slow.
- Teammate injury impact feature (medium-tier in V5_QUESTIONS.md) not
  implemented. Defer pending ablation results.
- add_opponent_defense_rank has triple nested loop. Works but could be
  vectorized via merge of precomputed rank table. Defer pending perf data.
- Production inference skeleton (building feature rows for upcoming games
  before they happen) is NOT in V5 feature engineer scope. Documented in
  engineer.py docstring. Will be a separate pipeline for Task 4.x.

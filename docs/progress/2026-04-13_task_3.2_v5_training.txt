Goal
----
Train the V5 prediction models: 54 ensembles = 27 (position, stat) pairs × 2
model types (StatPredictor regression + POBModel binary classification). Each
ensemble averages per-position algorithm subsets (QB/WR get 3, RB/TE get 4,
K gets 2, DST gets 3). Total 174 .joblib files + 54 meta JSON + 1 summary
CSV. Walk-forward eval on 2021-2024 produces honest MAE/AUC numbers for
Task 3.2b ablation baseline and Task 3.2c production retrain.

Approach
--------
1. Per-position algorithm selection (not uniform 4-algo-everywhere):
   - QB (3,181 rows, rich features): xgboost + lightgbm + catboost. RF drag.
   - RB (7,428 rows, interaction-heavy): all 4 algos; RF adds diversity.
   - WR (11,917 rows, high variance): xgboost + lightgbm + catboost.
   - TE (5,904 rows, sparse advanced stats): all 4 algos; LightGBM gets
     min_data_in_leaf=50 override to combat leaf-wise overfit.
   - K (2,715 rows, 3-stat sparse outcomes): xgboost + RF only. CatBoost/
     LightGBM minimum-sample overhead hurts at N=2700.
   - DST (2,686 rows, narrow 42-col features): xgboost + catboost + RF.
     LightGBM needs more data.
   Rationale: each algorithm has known strengths — matching to position
   data density + feature width maximizes signal while reducing compute
   (216 models → 174, ~20% fewer).

2. Per-stat loss function:
   - COUNT_STATS (TDs, INTs, receptions, targets, sacks, etc.): Poisson
     objective (xgboost count:poisson, lightgbm/catboost native Poisson).
     Calibrated predictions for right-skewed non-negative integer counts.
   - CONTINUOUS (yards, points_allowed): RMSE.
   - RF falls back to MSE on count stats (no Poisson support); documented
     in meta JSON under objective_per_algo.

3. Whitelist-based feature selection (data leakage prevention):
   V5 features parquet contains BOTH raw current-week stats (target_share,
   wopr, ngs_*, pfr_*, *_exp) AND their rolling_* equivalents. Naive
   loader would include the raw ones → massive leakage. Whitelist accepts
   only rolling_/variance_/trend_/opp_/prior_/is_ prefixes + 8 pre-game
   context columns (spread_line, total_line, team_rest, temp, wind, etc.).
   Regression test pins known leak columns against exclusion.

4. Walk-forward expanding window:
   For each (eval_season, eval_week) in 2021-2024, train on strictly-prior
   rows, predict the eval week. Mirrors production prediction semantics.
   2020 included in training (useful data) but excluded from eval (COVID
   noise per roadmap). Warm-up 2018-2019 dropped by history filter.

Key architecture decisions
--------------------------
- StatPredictor + POBModel (NOT EVOB — dropped from V5 per roadmap).
- Ensemble averaging = simple mean of per-position algo predictions.
- Atomic file writes: joblib.dump to .tmp → os.replace to final.
  .tmp orphan sweep at train_all start handles mid-write kills.
- Resume per-ensemble: `ensemble_files_complete` checks N algo files +
  meta JSON. Partial state (no meta) → retrain; complete state → skip.
- Meta JSON records feature_columns, neutral_fills (imputation policy),
  objective_per_algo, algorithms_used — Task 3.2c detects policy drift.
- _atomic_append_csv auto-rotates _mae_summary.csv on schema mismatch
  (prevents NaN-polluted mix of old + new column names).
- No StandardScaler (tree models don't need it; legacy V4 added complexity
  for zero benefit).
- K position applies drop_all_null_columns: 88 whitelist columns → 75
  after K-only row filter (passing/rushing rolling stats that K players
  never have are pruned).

Lessons learned
---------------
1. Four review rounds caught 3 critical bugs that single-pass review
   would have shipped. Each would have been silent in production:
   - Round 1: walk-forward index misalignment (eval_df/preds could
     shift if prepare() dropped NaN rows).
   - Round 2: predict-time fill_features skip — the dome-temp fix only
     applied at training. Inference would silently use 0°F for 36% of
     all predictions (every dome team's games).
   - Round 3: Colab Step 8 still referenced `df['mae']` after round-1
     rename to `mae_v5`. Would KeyError-crash after 1-3 hours of Colab
     training with no summary table output.
   The meta-lesson: each round found a bug a DIFFERENT class from the
   prior round. Single-pass review biases toward finding one class
   (usually correctness). Multi-pass with explicit different angles
   (correctness → integration → latent hazards → test quality) surfaces
   bugs that any single lens misses. The cost was ~45 min reviewer time
   for ~4 hours of saved production debugging.

2. Regression tests can have escape routes. Round 4 found that the
   test_predict_uses_fill_features_for_dome_temp_consistency test used
   rolling_avg_receptions ∈ [2, 8] uniform as a noise feature — tree
   models could pass the > 3.0 threshold via that correlation alone,
   masking the bug it was meant to catch. Fix: make noise features
   CONSTANT (5.0) so the model must use temp. Applies broadly: if a
   regression test doesn't fail when you break the behavior it's
   meant to guard, it's not load-bearing. Worth verifying by briefly
   reverting the fix and confirming the test fails.

3. "Same fix name but different code path" is a classic blind spot.
   NEUTRAL_FILLS was applied in _BaseEnsemble.fit via fill_features(X).
   But _predict_per_algo and POBModel.predict independently called
   X.fillna(0.0). The same dict was being SOURCED at the wrong call
   sites. Two reviewers in Round 2 independently caught this — it
   wouldn't have been caught by "the fit-side code works." Lesson:
   when fixing a train/predict consistency bug, grep for ALL call
   sites that touch the same data structure, not just the one you
   edited.

4. Per-position algorithm selection beats "use all 4 everywhere."
   LightGBM overfits on TE (~3K rows, leaf-wise growth aggressive).
   RF adds drag on QB (~3K rows, gradient boosters dominate). K at
   600 rows doesn't benefit from CatBoost/LightGBM's minimum-sample
   overhead. Dropping these reduced total model count 216 → 174 (~20%
   less compute) with no expected MAE loss. Cost: per-position config
   complexity in POSITION_ALGORITHMS dict. Benefit: ~20 min less Colab
   time per run + honest "I matched algorithm to data density" story
   for the portfolio.

5. Poisson loss for count stats matters. RMSE is symmetric around zero
   but count outcomes (TDs, INTs, sacks, receptions) are right-skewed
   and bounded below by 0. XGBoost's count:poisson, LightGBM's poisson,
   CatBoost's Poisson objectives all handle this natively. Expected
   5-15% MAE improvement on count stats vs RMSE baseline. RF doesn't
   support Poisson and falls back to MSE — documented per-algo in meta
   JSON so drift is traceable.

6. Whitelist > blacklist for leakage prevention. First pass used a
   blacklist of raw stat column names (passing_yards, rushing_yards,
   etc.). Real-data smoke revealed V5 features parquet contained ~20
   other current-week observable columns (target_share, wopr, ngs_*,
   *_exp) that weren't on the blacklist. TE/receptions MAE of 0.27 on
   real data was the red flag (should be 1.5-2.5 — the model was
   leaking). Switched to whitelist: only rolling_/variance_/trend_/
   opp_/prior_/is_ prefixes + explicit safe-context allowlist. MAE
   jumped to expected range. Lesson: leakage blacklists will always
   be incomplete; whitelist the known-safe shapes and fail-closed
   for everything else.

7. Dome weather imputation was 36% of all rows. NaN rate analysis
   (31% for temp, same for wind) revealed dome games systematically
   get weather = NaN → fillna(0) treats them as 0°F freezing
   conditions. Would silently bias passing predictions downward for
   8 dome teams across ~12 games/season each = ~96 games/season per
   team systematically mispredicted. NEUTRAL_FILLS["temp"] = 65.0
   fixes it. Lesson: always check NaN rate per column BEFORE defaulting
   to a single fillna strategy. 36% NaN is not "rare missing data" —
   it's a systematic pattern that demands per-column handling.

Remaining risks / Deferred
--------------------------
- DST sample size for early eval seasons. Walk-forward for 2021 W1
  trains on 2020 DST only (~512 rows). If MAE is suspiciously high
  for early 2021 DST weeks, consider restricting eval_seasons to
  [2023, 2024] only. Decision deferred to post-Colab review.

- V4 MAE comparison is apples-to-oranges. V4 baselines (5.14/4.66/
  4.66/4.26) used different splits + features. mae_v4 column NOT
  included in _mae_summary.csv. V4 cited as reference only in meta.
  Honest V4-vs-V5 comparison requires retraining V4 on V5's split —
  deferred to Task 3.2b or beyond.

- fillna(0) still applies to all non-temp context columns. If future
  NaN patterns emerge in rest days, spreads, or is_* flags, they'll
  silently fill as 0. Current NaN rates for those columns are 0%
  (verified during build phase). Revisit if MAE looks off for specific
  positions post-Colab.

- Task 3.2c needs to respect neutral_fills from meta JSON. Production
  retrain + prediction pipeline must detect if local NEUTRAL_FILLS
  differs from saved meta's neutral_fills dict. Contract is embedded
  in meta JSON but enforcement is deferred to 3.2c's load path.

- Per-algo prediction variance is not measured. Ensemble averaging
  could hide one algo silently producing bad predictions. Consider
  adding per-algo prediction variance to meta JSON as a "confidence"
  proxy. Deferred; revisit if portfolio demo needs uncertainty bands.

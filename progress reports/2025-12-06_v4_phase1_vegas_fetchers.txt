================================================================================
TECHNICAL DECISIONS
================================================================================

1. MODULAR ARCHITECTURE
   Why: Clean separation, easier testing, maintainability
   - v4_espnscraper.py: ESPN logic only
   - odds_api.py: The Odds API logic only
   - odds_fetcher.py: Orchestration only

   Benefit: Can test each component independently

2. HYBRID DATA APPROACH
   Why: Best of both worlds
   - ESPN: Historical data (2020-2024), unlimited, free
   - Odds API: Current data (2025), real-time, 500/month

   Benefit: Complete dataset without hitting API limits

3. SMART CACHING
   Why: Minimize API calls, faster subsequent runs
   - Historical: Cache forever (games are final)
   - Current: Can refresh (lines change)
   - Format: Parquet (fast, efficient)

   Benefit: First run takes 2-3 hours, future runs instant

4. ESPN UNOFFICIAL API
   Why: Fast, structured JSON, comprehensive
   - Endpoint: site.web.api.espn.com/apis/site/v2/sports/football/nfl
   - Data: Scores, stats, betting lines (pickcenter)
   - No auth required

   Risk: Unofficial, could change
   Mitigation: Can fallback to scraping ESPN pages

5. PARQUET FORMAT
   Why: Fast, efficient, industry standard
   - Columnar storage (faster queries)
   - Compression (smaller files)
   - Type preservation (dates, floats)

   Benefit: Instant loading vs CSV parsing
